# Project Context

## Purpose
Dashboard - Account Management System для управления аккаунтами с возможностью:
- Просмотра и редактирования данных аккаунтов
- Фильтрации и сортировки записей
- Массового обновления статусов
- Экспорта данных в CSV
- Массового переноса аккаунтов между статусами
- Детального просмотра информации об аккаунте

**Целевая аудитория:** Внутренние пользователи для управления базой аккаунтов

## Tech Stack

### Backend
- **PHP 7.3+** (рекомендуется PHP 8.0+)
- **MySQL 5.7+ / MariaDB 10.2+** (InnoDB engine)
- Архитектура: MVC-подобная, сервис-ориентированная
- ORM: Нет (используется чистый PDO/mysqli с prepared statements)

### Frontend
- **HTML5, CSS3**
- **JavaScript (Vanilla JS)** - ООП подход (класс Dashboard)
- **Bootstrap 5.3** - UI фреймворк
- **Font Awesome 6.4** - иконки
- Без frontend фреймворков (React/Vue/Angular)

### Database
- **MySQL/MariaDB** с кодировкой utf8mb4
- Основная таблица: `accounts` (20+ колонок)
- Используются индексы для оптимизации запросов

### Security
- Prepared statements для защиты от SQL injection
- CSRF токены для защиты форм
- XSS защита через `htmlspecialchars()`
- Хэширование паролей через `password_hash()` (bcrypt)
- HTTP Security Headers в .htaccess

## Project Conventions

### Code Style

#### PHP
- **Отступы:** 4 пробела (не табы)
- **Неймінг:**
  - Классы: `PascalCase` (AccountsService, FilterBuilder)
  - Методы: `camelCase` (getAccounts, createFilter)
  - Переменные: `snake_case` для переменных БД, `camelCase` для остального
  - Константы: `UPPER_CASE` (DB_HOST, MAX_REQUEST_SIZE)
- **Комментарии:** На русском языке для важных блоков кода
- **Структура файлов:**
  - Один класс = один файл
  - Имя файла = имя класса

#### JavaScript
- **Отступы:** 4 пробела
- **Неймінг:**
  - Классы: `PascalCase` (Dashboard)
  - Методы: `camelCase` (handleCellEdit, applyFilter)
  - Константы: `UPPER_CASE`
- **ООП подход:** Использование ES6 классов
- **Event delegation:** Предпочтительнее множественных обработчиков

#### CSS
- **Отступы:** 4 пробела
- **Неймінг:** kebab-case (.filter-panel, .table-row)
- **Организация:** Модульные файлы (dashboard.css, dashboard-main.css, dashboard-pro.css)

### Architecture Patterns

#### Слои приложения
```
Presentation Layer (UI)
    ↓
Controller Layer (Entry Points)
    ↓
Service Layer (Business Logic)
    ↓
Data Access Layer (Database)
```

#### Основные паттерны

1. **Service Layer Pattern**
   - `AccountsService` - единая точка доступа к данным аккаунтов
   - `MassTransferService` - массовый перенос аккаунтов
   - Вся бизнес-логика находится в сервисах

2. **Singleton Pattern**
   - `Database::getInstance()` - единственное подключение к БД
   - Lazy loading подключения

3. **Builder Pattern**
   - `FilterBuilder` - построение SQL фильтров
   - Fluent interface: `$filter->addSearchFilter()->addStatusFilter()`

4. **Repository Pattern (частично)**
   - Сервисы инкапсулируют доступ к данным
   - Prepared statements для всех запросов

#### Структура директорий
```
dashboard/

│   ├── FilterBuilder.php

│   ├── MassTransferService.php
│   └── Utils.php
├── templates/             # HTML шаблоны
│   └── dashboard.php
├── assets/                # Статические ресурсы
│   ├── css/

```

#### Принципы
- **Separation of Concerns**: Разделение логики, данных и представления
- **DRY (Don't Repeat Yourself)**: Избегаем дублирования кода
- **Security First**: Всегда используем prepared statements
- **Кэширование**: Кэш метаданных БД в `/tmp` на 1 час
- **Минимализм**: Простые решения вместо сложных фреймворков

### Git Workflow

**Стратегия веток:**
- `main` / `master` - production-ready код
- Feature branches для новых функций
- Hotfix branches для срочных исправлений

**Commit conventions:**
- На русском языке
- Формат: `[тип] краткое описание`
- Типы: `feat`, `fix`, `refactor`, `docs`, `style`, `perf`, `test`
- Примеры:
  - `feat: добавлена функция массового экспорта`
  - `fix: исправлена ошибка фильтрации по статусу`
  - `refactor: оптимизация запросов статистики`

**Важно:**
- Никогда не использовать `--force` для main/master
- Не пропускать pre-commit hooks
- Код-ревью обязательно для важных изменений

## Domain Context

### Основные сущности

#### Account (Аккаунт)
Основная сущность системы, содержит:
- **Учетные данные:** login, password, email, email_password
- **Личные данные:** first_name, last_name, birth_day, birth_month, birth_year
- **Социальные данные:** social_url, ads_id
- **Техническая информация:** token, cookies, user_agent, two_fa
- **Метаданные:** status, created_at, updated_at
- **Дополнительные поля:** extra_info_1-4 (назначение зависит от бизнес-логики)

#### Status (Статус)
Свободное текстовое поле для категоризации аккаунтов. Примеры статусов:
- Активные/неактивные аккаунты
- По стадиям обработки
- По типам аккаунтов
- Может быть пустым (NULL или '')

#### Filters (Фильтры)
Система поддерживает множественные фильтры:
- **Поиск:** по всем текстовым полям
- **По статусу:** множественный выбор
- **По наличию данных:** has_email, has_2fa, has_token и т.д.
- **По диапазонам:** для числовых полей
- **По датам:** created_at, updated_at

### Бизнес-процессы

1. **Просмотр аккаунтов:** Пагинация, сортировка, фильтрация
2. **Редактирование:** Inline-редактирование полей прямо в таблице
3. **Массовые операции:**
   - Изменение статуса для выбранных аккаунтов
   - Удаление множества аккаунтов
   - Массовый перенос между статусами
4. **Экспорт:** CSV экспорт с учётом текущих фильтров
5. **Статистика:** Общее количество, распределение по статусам, изменения за 24 часа

## Important Constraints

### Технические ограничения

1. **Хостинг:** Shared hosting с ограниченными ресурсами
   - Нельзя использовать Node.js/Python
   - Ограниченная память PHP (часто 128-256MB)
   - Нет доступа к системным командам
   - Нет возможности установки расширений PHP

2. **База данных:**
   - MySQL/MariaDB (не PostgreSQL)
   - Нет доступа к stored procedures в некоторых случаях
   - Ограничение на размер запроса
   - Ограничение на количество одновременных подключений

3. **Файловая система:**
   - Хранение пользователей в users.json (не масштабируемо)
   - Кэш в /tmp может очищаться хостингом
   - Ограничения на запись в директории

4. **Производительность:**
   - Без Redis/Memcached на текущем окружении
   - In-memory кэш теряется между запросами
   - Большие экспорты могут превышать timeout PHP

5. **Безопасность:**
   - .htaccess для защиты конфиденциальных файлов
   - Нет возможности использовать современные инструменты безопасности
   - Ограничения на заголовки безопасности

### Бизнес-ограничения

1. **Совместимость:**
   - Должно работать на старых версиях PHP (7.3+)
   - Поддержка IE11 не требуется
   - Мобильная версия - базовая поддержка

2. **Масштабирование:**
   - Целевой объем: до 100,000 аккаунтов
   - Одновременных пользователей: до 10
   - Не требуется высокая доступность (99.99%)

3. **Регуляторные требования:**
   - Хранение паролей в хэшированном виде (bcrypt)
   - Логирование критических операций
   - Защита персональных данных

### Архитектурные ограничения

1. **Не использовать:**
   - Composer (сложности с хостингом)
   - Frontend фреймворки (React/Vue/Angular)
   - Build инструменты (Webpack/Vite) - только ванильный JS
   - Сложные ORM (Doctrine/Eloquent)

2. **Предпочитать:**
   - Простые решения вместо сложных
   - Нативные возможности PHP
   - Минимальные зависимости
   - Файлы без минификации (для простоты отладки)

## Известные проблемы и технический долг

### Критический технический долг
1. **Дублирование кода:** ~15% кода дублируется (особенно логика фильтрации)
2. **Файловое хранилище пользователей:** users.json не масштабируется
3. **N+1 проблема:** 8 запросов к БД для статистики (должен быть 1)
4. **Широкая таблица:** accounts имеет 20+ колонок (нужна нормализация)

### Планируемые улучшения
- Миграция users.json в БД
- Оптимизация запросов статистики
- Добавление Rate Limiting для API
- Нормализация схемы БД
- Добавление unit тестов (цель: 70% покрытие)

## Процесс разработки с OpenSpec

### Когда создавать proposal
- Новые функции (новые страницы, API endpoints)
- Изменения схемы БД
- Рефакторинг архитектуры
- Оптимизации, влияющие на производительность
- Изменения безопасности

### Когда НЕ нужен proposal
- Исправление багов (возврат к задуманному поведению)
- Опечатки, форматирование
- Обновление комментариев
- Мелкие улучшения UI без изменения логики
- Добавление логов/отладочной информации

### Naming conventions для changes
- `add-[feature]` - новая функция
- `update-[feature]` - изменение существующей функции
- `remove-[feature]` - удаление функции
- `refactor-[component]` - рефакторинг
- `fix-[issue]` - критическое исправление (если требует архитектурных изменений)

Примеры:
- `add-two-factor-auth`
- `update-export-performance`
- `refactor-database-service`
- `remove-legacy-index-php`
