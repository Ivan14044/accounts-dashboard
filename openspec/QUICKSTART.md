# 🚀 OpenSpec - Быстрый старт для Dashboard

## Что изменилось в работе?

### ДО OpenSpec 😰
```
Вы: "Добавь функцию экспорта в PDF"
AI: *начинает кодить сразу*
    *меняет 5 файлов*
    *что-то ломает*
    *нет документации*
    *непонятно, что именно изменилось*
```

### ПОСЛЕ OpenSpec 😎
```
Вы: "Добавь функцию экспорта в PDF"
AI: *создаёт proposal с описанием*
    *определяет затронутые части системы*
    *создаёт пошаговый план*
    *валидирует изменения*
    *реализует по плану*
    *всё задокументировано*
```

## 🎯 Основные сценарии использования

### Сценарий 1: Простое изменение (без proposal)

```
Вы: "Исправь опечатку в файле login.php"
AI: *исправляет напрямую*
    (proposal не нужен для мелких правок)
```

**Не требуют proposal:**
- Исправление багов
- Опечатки
- Обновление комментариев
- Мелкие UI правки

### Сценарий 2: Крупное изменение (с proposal)

```
Вы: "Добавь функцию массового импорта аккаунтов из CSV"
AI: "Это крупное изменение. Создам proposal..."
    
    1. Создаёт openspec/changes/add-csv-import/
       - proposal.md (зачем, что, влияние)
       - tasks.md (пошаговый план)
       - specs/data-import/spec.md (требования)
    
    2. Показывает вам для одобрения
    
    3. После одобрения реализует по плану
    
    4. После деплоя архивирует
```

**Требуют proposal:**
- Новые функции
- Изменения архитектуры
- Изменения схемы БД
- Оптимизации производительности
- Breaking changes

## 📋 Практические примеры

### Пример 1: Добавление фильтра по дате

**Ваш запрос:**
```
"Добавь возможность фильтровать аккаунты по дате создания 
(диапазон от-до)"
```

**Что сделает AI:**

1. **Создаст proposal:**
```markdown
# Change: Фильтр по дате создания

## Why
Пользователи хотят видеть аккаунты, созданные в определённый период.

## What Changes
- Добавление полей "Дата от" и "Дата до" в фильтры
- Обновление FilterBuilder для поддержки date_range
- Обновление UI с datepicker
- Добавление индекса на created_at

## Impact
- Affected specs: account-management
- Affected code: FilterBuilder.php, dashboard.php, dashboard.js
```

2. **Создаст tasks.md:**
```markdown
## 1. Backend
- [ ] 1.1 Добавить метод addDateRangeFilter в FilterBuilder
- [ ] 1.2 Обновить AccountsService::createFilterFromRequest

## 2. Frontend
- [ ] 2.1 Добавить поля date_from и date_to в HTML
- [ ] 2.2 Интегрировать datepicker
- [ ] 2.3 Обработать в Dashboard.applyFilter()

## 3. Database
- [ ] 3.1 Проверить индекс на created_at
```

3. **Создаст спецификацию:**
```markdown
## ADDED Requirements
### Requirement: Date Range Filter
Система ДОЛЖНА позволять фильтровать аккаунты по диапазону дат создания.

#### Scenario: Фильтрация с обеими датами
- **WHEN** пользователь указывает дату от 01.01.2024 и дату до 31.01.2024
- **THEN** система показывает только аккаунты, созданные в этом диапазоне

#### Scenario: Только дата от
- **WHEN** пользователь указывает только дату от 01.01.2024
- **THEN** система показывает аккаунты, созданные после 01.01.2024
```

4. **Покажет вам для одобрения**
5. **Реализует пошагово**
6. **Отметит выполненные задачи в tasks.md**

### Пример 2: Оптимизация запросов

**Ваш запрос:**
```
"Дашборд медленно загружается. Оптимизируй запросы к БД."
```

**Что сделает AI:**

1. **Анализирует проблему:**
   - Изучает текущие запросы
   - Находит N+1 проблему (8 запросов вместо 1)
   
2. **Создаёт proposal:**
```markdown
# Change: Оптимизация запросов статистики

## Why
Загрузка дашборда занимает ~800ms из-за множественных запросов к БД.

## What Changes
- Объединение 8 запросов в 1 через агрегацию
- Добавление недостающих индексов
- Кэширование результатов

## Impact
- Affected specs: account-management
- Affected code: AccountsService::getStatistics()
- Performance: 800ms → ~150ms (5x улучшение)
```

3. **Реализует с сохранением старого функционала**

**Что сделает AI:**
```
AI: "Это баг - исправляю напрямую без proposal"
    *находит проблему в FilterBuilder*
    *исправляет логику*
    *тестирует*
    
    (proposal не нужен для багфиксов)
```

## 🔍 Как работать с AI теперь

### 1. Формулируйте запросы как обычно

Вам НЕ нужно:
- Говорить "создай proposal"
- Упоминать OpenSpec
- Менять стиль общения

AI сам определит, нужен ли proposal.

### 2. Одобряйте крупные изменения

Когда AI создаст proposal, он покажет вам:
```
📋 Создан proposal: add-csv-import

Файлы:
- openspec/changes/add-csv-import/proposal.md
- openspec/changes/add-csv-import/tasks.md
- openspec/changes/add-csv-import/specs/data-import/spec.md

Пожалуйста, ознакомьтесь. Начать реализацию?
```

Вы можете:
- ✅ Одобрить: "Да, реализуй"
- ✏️ Изменить: "Измени tasks.md - добавь валидацию"
- ❌ Отклонить: "Нет, это не нужно"

### 3. Следите за прогрессом

AI автоматически:
- Отмечает выполненные задачи в `tasks.md`
- Обновляет статусы
- Документирует изменения

### 4. После деплоя

Попросите AI:
```
"Заархивируй change add-csv-import"
```

AI выполнит:
```bash
openspec archive add-csv-import --yes
```

## 📊 Реальные преимущества

### Преимущество 1: Прозрачность

**ДО:**
- Не знаешь, что AI изменит
- Сюрпризы после реализации
- Сложно отследить изменения

**ПОСЛЕ:**
- Чёткий план ДО реализации
- Видно влияние на систему
- История всех изменений

### Преимущество 2: Безопасность

**ДО:**
- AI может случайно сломать старый функционал
- Нет проверки конфликтов

**ПОСЛЕ:**
- AI видит, какие части затрагиваются
- Проверка breaking changes
- Валидация перед реализацией

### Преимущество 3: Документация

**ДО:**
- Код без документации
- Непонятно, что делает функция
- Сложно вспомнить через месяц

**ПОСЛЕ:**
- Каждое изменение задокументировано
- Есть требования и сценарии
- История изменений в archive/

### Преимущество 4: Контроль качества

**ДО:**
- AI может сделать "как попроще"
- Нет проверки корректности

**ПОСЛЕ:**
- Автоматическая валидация: `openspec validate --strict`
- Чёткие требования к реализации
- Проверка полноты сценариев

## 🎓 Команды для вас

Хотя AI делает всё автоматически, вы можете использовать:

```bash
# Посмотреть активные изменения
openspec list

# Посмотреть существующие спецификации
openspec list --specs

# Детали конкретного изменения
openspec show add-csv-import

# Проверить корректность
openspec validate --strict

# Заархивировать после деплоя
openspec archive add-csv-import --yes
```

## 🚨 Частые вопросы

### Q: Нужно ли мне учить новые команды?
**A:** Нет! Общайтесь с AI как обычно. Он сам использует OpenSpec.

### Q: Замедлит ли это разработку?
**A:** Для мелких правок - нет (они идут напрямую).  
Для крупных изменений - наоборот, ускорит (меньше багов, чёткий план).

### Q: Что если я хочу просто "быстро попробовать"?
**A:** Скажите: "Сделай быстрый прототип без proposal".  
AI пропустит proposal для эксперимента.

### Q: Можно ли изменить proposal после создания?
**A:** Да! Просто скажите: "Измени tasks.md - добавь пункт X".

### Q: Нужно ли одобрять каждое изменение?
**A:** Только крупные (новые функции, архитектура).  
Багфиксы и мелкие правки идут автоматически.
