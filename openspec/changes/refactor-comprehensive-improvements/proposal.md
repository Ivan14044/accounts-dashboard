# Change: Комплексные улучшения проекта Dashboard

## Why

Проект имеет хорошую архитектурную основу (оценка 7.5/10 по техническому аудиту), но имеет ряд проблем, которые снижают производительность, безопасность и удобство поддержки:

1. **Производительность:** 8 запросов к БД для статистики вместо 1 (800ms загрузка)
2. **Безопасность:** Избыточное логирование чувствительных данных, нет rate limiting
3. **Дублирование кода:** ~15% кода дублируется (особенно в export.php)
4. **Масштабируемость:** Файловое хранилище users.json не масштабируется
5. **Читабельность:** Magic numbers вместо констант
6. **UX:** Используется устаревший alert() вместо современных уведомлений

## What Changes

### Фаза 1: Quick Wins (Критические, 1-2 дня)

#### 1.1 Безопасность - Удаление избыточного логирования
- Удалить debug логирование из `status_update.php`, `delete.php`, `mass_transfer.php`
- Создать класс `Logger` с уровнями логирования (debug, info, error)
- Использовать `Logger::debug()` вместо прямого `error_log()`
- **Выигрыш:** Безопасность (не логируем CSRF токены, пароли), меньше логов на диске

#### 1.2 Чистота кода - Рефакторинг export.php
- Удалить дублирующую логику фильтрации (~150 строк)
- Использовать `AccountsService::createFilterFromRequest()`
- Использовать единообразную обработку экспорта
- **Выигрыш:** -150 строк кода, единообразие, проще поддержка

#### 1.3 Производительность - Оптимизация запросов статистики
- Объединить 8 запросов в 1 через агрегацию SQL
- Использовать `JSON_OBJECTAGG` для группировки по статусам
- Кэшировать результат на 5 минут
- **Выигрыш:** 8x ускорение (800ms → ~150ms)

#### 1.4 Производительность - Индексы БД
- Добавить индекс `idx_email_status` на `(email, status)`
- Добавить индекс `idx_two_fa_not_empty` на `two_fa`
- Добавить индекс `idx_token_not_empty` на `token`
- **Выигрыш:** 2-5x ускорение запросов с фильтрами

#### 1.5 Читабельность - Класс Config с константами
- Создать `includes/Config.php` с константами
- Заменить magic numbers на именованные константы
- Константы для лимитов, таймаутов, размеров
- **Выигрыш:** Читабельность кода, легче настройка

### Фаза 2: Средний приоритет (1 неделя)

#### 2.1 Безопасность - Rate Limiting
- Создать класс `RateLimiter` с использованием файлового кэша
- Добавить проверку лимитов в API endpoints
- Конфигурируемые лимиты (100 req/min по умолчанию)
- **Выигрыш:** Защита от brute-force и DoS атак

#### 2.2 Масштабируемость - Миграция users.json в БД
- Создать таблицу `users` в БД
- Создать таблицу `user_sessions` для сессий
- Миграция данных из users.json
- Обновить auth.php для работы с БД
- **Выигрыш:** Масштабируемость, безопасность, аудит

#### 2.3 UX - Toast notifications
- Создать класс `Toast` в JavaScript
- Заменить `alert()` на `Toast.show()`
- Современные уведомления с автоскрытием
- **Выигрыш:** Современный UX, не блокирует интерфейс
