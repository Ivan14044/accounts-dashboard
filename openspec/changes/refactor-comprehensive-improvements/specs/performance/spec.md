# Spec Delta: Performance

## ADDED Requirements

### Requirement: Индексы БД для оптимизации запросов
Система SHALL иметь индексы на часто используемых полях для ускорения запросов с фильтрами.

#### Scenario: Индекс на email и status
- **WHEN** пользователь фильтрует аккаунты по email и статусу
- **THEN** запрос использует индекс `idx_email_status`
- **AND** выполняется в 2-5x быстрее чем без индекса

#### Scenario: Индекс на two_fa для фильтра "есть 2FA"
- **WHEN** пользователь применяет фильтр "Есть 2FA"
- **THEN** запрос использует индекс `idx_two_fa_not_empty`
- **AND** быстро находит записи с непустым two_fa

#### Scenario: Индекс на token для фильтра "есть токен"
- **WHEN** пользователь применяет фильтр "Есть токен"
- **THEN** запрос использует индекс `idx_token_not_empty`
- **AND** эффективно фильтрует записи

#### Scenario: Автоматическое создание индексов
- **WHEN** система запускается впервые после обновления
- **THEN** метод `Database::ensureIndexes()` проверяет наличие индексов
- **AND** создаёт отсутствующие индексы автоматически
- **AND** логирует создание индексов

### Requirement: Кэширование метаданных БД
Система SHALL кэшировать метаданные таблиц для уменьшения количества запросов `SHOW COLUMNS`.

**Статус:** Уже реализовано в `ColumnMetadata`, улучшено использованием `Config::CACHE_TTL`.

#### Scenario: Использование кэшированных метаданных
- **WHEN** система запрашивает структуру таблицы
- **THEN** проверяет кэш в `/tmp`
- **AND** если кэш свежий (< 1 часа), использует его
- **AND** не выполняет `SHOW COLUMNS`

#### Scenario: Обновление кэша метаданных
- **WHEN** кэш устарел или отсутствует
- **THEN** выполняет `SHOW COLUMNS FROM accounts`
- **AND** сохраняет результат в кэш
- **AND** устанавливает TTL согласно `Config::CACHE_TTL`

### Requirement: Оптимизированная статистика через агрегацию
Система SHALL получать статистику аккаунтов одним SQL запросом вместо нескольких.

#### Scenario: Единый запрос статистики
- **WHEN** дашборд загружается
- **THEN** выполняется ОДИН SQL запрос с агрегацией
- **AND** использует `JSON_OBJECTAGG` для группировки по статусам
- **AND** использует `SUM(CASE WHEN ...)` для условного подсчёта
- **AND** время выполнения <150ms

#### Scenario: Кэширование результатов статистики
- **WHEN** статистика запрашивается
- **THEN** проверяется кэш с TTL = 5 минут
- **AND** если кэш актуален, возвращается из кэша (<10ms)
- **AND** если кэш устарел, выполняется запрос и обновляется кэш

#### Scenario: Инвалидация кэша при изменениях
- **WHEN** изменяются данные аккаунтов (статус, удаление, создание)
- **THEN** кэш статистики инвалидируется
- **AND** следующий запрос получает свежие данные

### Requirement: Минимизация запросов к БД
Система SHALL минимизировать количество запросов к БД при загрузке дашборда.

#### Scenario: Загрузка дашборда с минимальными запросами
- **WHEN** пользователь открывает дашборд
- **THEN** выполняется максимум 5 запросов к БД:
  1. Метаданные таблицы (если не в кэше)
  2. Статистика (один агрегированный запрос)
  3. Данные аккаунтов (с пагинацией и фильтрами)
  4. Подсчёт общего количества (если применён фильтр)
  5. Опциональные дополнительные запросы
- **AND** общее время запросов <200ms

#### Scenario: Использование prepared statements для производительности
- **WHEN** выполняются повторяющиеся запросы
- **THEN** используются prepared statements
- **AND** MySQL может кэшировать план выполнения
- **AND** запросы выполняются быстрее

## Performance Targets

### Метрики до оптимизации:
- Загрузка дашборда: ~800ms
- Запросов к БД при загрузке: 8-12
- Запрос статистики: 8 отдельных запросов, ~300ms
- Запросы с фильтрами: без индексов, медленно на >10K записей

### Метрики после оптимизации:
- Загрузка дашборда: <200ms (✅ 4x улучшение)
- Запросов к БД при загрузке: 3-5 (✅ 2.4x уменьшение)
- Запрос статистики: 1 агрегированный запрос, <150ms (✅ 8x улучшение)
- Запросы с фильтрами: с индексами, 2-5x быстрее (✅)

### Acceptance Criteria:
- ✅ Загрузка дашборда ДОЛЖНА быть <200ms для 10,000 записей
- ✅ Запрос статистики ДОЛЖЕН быть <150ms
- ✅ Повторный запрос статистики ДОЛЖЕН быть <10ms (из кэша)
- ✅ Количество запросов к БД ДОЛЖНО быть 3-5 при загрузке
- ✅ Запросы с фильтрами ДОЛЖНЫ использовать индексы
