# Spec Delta: Security

## ADDED Requirements

### Requirement: Rate Limiting для API endpoints
Система SHALL ограничивать количество запросов к API endpoints для защиты от брутфорса и DoS атак.

#### Scenario: Нормальное использование в пределах лимита
- **WHEN** пользователь выполняет 50 запросов в минуту к API
- **THEN** все запросы обрабатываются успешно
- **AND** заголовок `X-RateLimit-Remaining` показывает оставшиеся запросы

#### Scenario: Превышение лимита запросов
- **WHEN** пользователь выполняет более 100 запросов в минуту к обычному API
- **THEN** система возвращает HTTP 429 Too Many Requests
- **AND** заголовок `Retry-After` указывает время ожидания в секундах
- **AND** JSON ответ содержит сообщение об ограничении

#### Scenario: Разные лимиты для разных endpoints
- **WHEN** пользователь обращается к endpoint экспорта
- **THEN** применяется лимит 10 запросов в минуту (более строгий)
- **WHEN** пользователь обращается к обычному API
- **THEN** применяется лимит 100 запросов в минуту

#### Scenario: Лимит на login endpoint
- **WHEN** пользователь пытается войти более 5 раз в минуту
- **THEN** система блокирует попытки входа на 1 минуту
- **AND** логирует подозрительную активность
- **AND** защищает от брутфорс атак

### Requirement: Уровневая система логирования
Система SHALL использовать уровни логирования и не логировать чувствительные данные в production.

#### Scenario: Debug логирование в development
- **WHEN** переменная окружения `DEBUG=true`
- **THEN** система логирует debug сообщения
- **AND** помогает в отладке

#### Scenario: Отключение debug в production
- **WHEN** переменная окружения `DEBUG` не установлена или `false`
- **THEN** debug сообщения НЕ логируются
- **AND** не создают избыточных логов
- **AND** не влияют на производительность

#### Scenario: Логирование ошибок без чувствительных данных
- **WHEN** происходит ошибка в API endpoint
- **THEN** система логирует сообщение об ошибке
- **AND** НЕ логирует CSRF токены
- **AND** НЕ логирует пароли
- **AND** НЕ логирует session ID
- **AND** НЕ логирует полные данные запроса

#### Scenario: Контекстное логирование
- **WHEN** система логирует событие
- **THEN** может включать контекст (user_id, action)
- **AND** контекст НЕ содержит чувствительных данных

## MODIFIED Requirements

### Requirement: Аутентификация пользователей
Система SHALL хранить пользователей в БД вместо файлового хранилища для масштабируемости.

**Изменение:** Миграция с users.json на таблицу `users` в БД.

#### Scenario: Аутентификация через БД
- **WHEN** пользователь вводит логин и пароль
- **THEN** система проверяет учетные данные в таблице `users`
- **AND** использует `password_verify()` для проверки хэша
- **AND** обновляет `last_login` timestamp

#### Scenario: Создание пользователя в БД
- **WHEN** администратор создаёт нового пользователя
- **THEN** система добавляет запись в таблицу `users`
- **AND** хэширует пароль через `password_hash()`
- **AND** устанавливает роль по умолчанию

#### Scenario: Управление сессиями в БД
- **WHEN** пользователь успешно аутентифицируется
- **THEN** система создаёт запись в таблице `user_sessions`
- **AND** сохраняет IP адрес и user agent
- **AND** устанавливает время истечения

#### Scenario: Валидация сессии
- **WHEN** пользователь делает запрос с session ID
- **THEN** система проверяет наличие сессии в `user_sessions`
- **AND** проверяет, что сессия не истекла
- **AND** при истечении удаляет сессию и требует повторного входа

#### Scenario: Fallback на users.json при проблемах с БД
- **WHEN** подключение к БД недоступно
- **THEN** система автоматически переключается на users.json
- **AND** логирует предупреждение о fallback
- **AND** пользователь может войти без прерывания сервиса

#### Scenario: Миграция данных из users.json
- **WHEN** администратор запускает скрипт миграции
- **THEN** система создаёт backup users.json.backup
- **AND** создаёт таблицы `users` и `user_sessions`
- **AND** переносит всех пользователей из JSON в БД
- **AND** проверяет успешность миграции

#### Scenario: Rollback миграции
- **WHEN** миграция вызвала проблемы
- **THEN** администратор может запустить скрипт отката
- **AND** восстанавливается users.json из backup
- **AND** удаляются таблицы users, user_sessions
- **AND** система возвращается к работе с JSON
