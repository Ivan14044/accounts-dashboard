## 1. CRITICAL — Немедленное исправление {#1-critical}

### C-01. Хардкод DB-кредов в исходниках
- **Файл:** `create_tables.php` (строки 7–22)
- **Проблема:** Логин, пароль и хост MySQL прописаны прямо в коде (`host: if592995.mysql.tools`, `password: zhA4*4@u8S`).
- **Риск:** Любой, кто имеет доступ к репо/исходникам, получает полный доступ к БД.
- **Фикс:** Переместить в `.env` / переменные окружения. Добавить файл в `.gitignore`.

### C-02. Хардкод FTP-кредов
- **Файл:** `deploy.php` (строки 7–9)
- **Проблема:** FTP-сервер, логин и пароль в открытом виде.
- **Риск:** Полный доступ к серверу через FTP.
- **Фикс:** Переместить в `.env`, убрать из Git-истории.

### C-03. Setup/Deploy скрипты без аутентификации
- **Файлы:** `auto_setup.php`, `create_tables.php`, `setup_favorites.php`, `setup_trash.php`, `apply_indexes_safe.php`, `create_optimization_flag.php`, `deploy.php`, `check_login_index.php`, `build_assets.php`, `_cleanup.php`, `download_account_template.php`
- **Проблема:** Ни один из этих скриптов не требует `requireAuth()`. Их можно вызвать напрямую через браузер.
- **Риск:** Анонимный пользователь может выполнить миграции, пересоздать таблицы, задеплоить код.
- **Фикс:** Добавить `requireAuth()` + проверку роли admin, или IP-whitelist.

### C-04. Дефолтный пароль admin123
- **Файл:** `auth.php` (строки 28–32, 48–52, 60–64)
- **Проблема:** При создании `users.json` используется захардкоженный пароль `admin123` (три дублирующихся блока).
- **Риск:** Предсказуемый пароль для первого входа.
- **Фикс:** Генерировать случайный пароль при первом запуске и показывать его один раз.

### C-05. Unsafe `unserialize()` в RateLimiter
- **Файл:** `includes/RateLimiter.php` (строка 197)
- **Проблема:** `@unserialize($content)` — если атакующий получит доступ к файловой системе, он может выполнить произвольный PHP-код через Object Injection.
- **Фикс:** Заменить на `json_encode()` / `json_decode()`.

### C-06. Undefined function `qprep()` — Fatal Error
- **Файл:** `includes/MassTransferService.php` (строки 152, 191)
- **Проблема:** Функция `qprep()` вызывается, но нигде не определена. Это **100% fatal error** при использовании mass transfer.
- **Фикс:** Заменить на стандартный `$mysqli->prepare()` + `bind_param()`.

### C-07. XSS через innerHTML в JavaScript
- **Файлы:** `assets/js/dashboard.js` (строки ~357, ~814), `assets/js/table-module.js` (строки 145, 159, 164, 528, 532, 544)
- **Проблема:** Данные из БД вставляются в DOM через `innerHTML` без экранирования.
- **Риск:** Если в поле аккаунта содержится `<script>`, он выполнится в браузере.
- **Фикс:** Использовать `textContent` или DOMPurify.

### C-08. CSRF-уязвимость: дубликат аккаунта через GET
- **Файл:** `view.php` (строки 623–667)
- **Проблема:** `duplicateAccount()` использует GET-запрос для изменения данных, без CSRF-токена.
- **Риск:** Атакующий может заставить пользователя создать дубликат аккаунта, подсунув ссылку.
- **Фикс:** Перевести на POST + CSRF token.

## 2. HIGH — Высокий приоритет {#2-high}

### H-01. DB-креды в сессии в открытом виде
- **Файл:** `auth.php` (строка 306)
- **Проблема:** `$_SESSION['db_config']` хранит логин/пароль БД plain text.
- **Фикс:** Шифровать перед сохранением или не хранить пароль в сессии.

### H-02. Race Condition: TOCTOU при проверке дубликатов логинов
- **Файл:** `includes/AccountsRepository.php` (строки 1111–1112)
- **Проблема:** Между проверкой `getExistingLogins()` и INSERT может прийти параллельный запрос с тем же логином.
- **Фикс:** UNIQUE INDEX на поле login + обработка `Duplicate entry`.

### H-03. Транзакции не закрываются при ошибке bind_param
- **Файл:** `includes/AccountsRepository.php` (строки 572–613)
- **Проблема:** Если `bind_param()` выбросит исключение, `rollback()` может не вызваться.
- **Фикс:** Обернуть весь блок в try/catch с гарантированным rollback в finally.

### H-04. Header Injection через ETag
- **Файл:** `includes/ResponseHeaders.php` (строка 55)
- **Проблема:** ETag вставляется в заголовок без санитизации. Если содержит `\r\n`, атакующий может вставить произвольные заголовки.
- **Фикс:** Удалять `\r`, `\n`, `"` из ETag перед вставкой.

### H-05. Session Fixation через X-Forwarded-Proto
- **Файл:** `includes/SessionManager.php` (строка 60)
- **Проблема:** Заголовок `HTTP_X_FORWARDED_PROTO` принимается без проверки. Атакующий может подменить его и сбросить флаг `secure` у cookie.
- **Фикс:** Принимать только от доверенных прокси (whitelist IP).

### H-06. Информация о БД в сообщениях об ошибках
- **Файл:** `config.php` (строки 96, 107)
- **Проблема:** В error-сообщения попадают хост, порт, имя пользователя БД.
- **Фикс:** Логировать детали, но возвращать пользователю только общую ошибку.

### H-07. Непоследовательная CSRF-валидация
- **Файлы:** `delete_permanent.php` (строка 31), `empty_trash.php` (строка 60)
- **Проблема:** Используют `verifyCsrfToken()` вместо `Validator::validateCsrfToken()`, как остальной код.
- **Фикс:** Унифицировать метод валидации CSRF по всему проекту.

### H-08. CSRF отсутствует на logout
- **Файл:** `logout.php`
- **Проблема:** Нет CSRF-токена. Атакующий может разлогинить пользователя через подставную ссылку.
- **Фикс:** Добавить CSRF-проверку или перевести на POST.

### H-09. Массовая ошибка подавления через `@`
- **Файлы:** `auto_setup.php`, `apply_indexes_safe.php`, `includes/ColumnMetadata.php`
- **Проблема:** Повсеместное использование `@` скрывает реальные ошибки.
- **Фикс:** Убрать `@`, использовать try/catch + Logger.

### H-10. Отсутствие авторизации на favorites
- **Файл:** `api/index.php` (строки 435–550)
- **Проблема:** При удалении favorite нет проверки, что он принадлежит текущему пользователю.
- **Фикс:** Добавить WHERE user_id = ? в запрос.

### H-11. DashboardController: apply_status через GET без CSRF
- **Файл:** `includes/DashboardController.php` (строки 28–76)
- **Проблема:** Изменение статуса аккаунтов через GET-параметр без CSRF.
- **Фикс:** POST + CSRF.

### H-12. Case-sensitive проверка дубликатов логинов
- **Файл:** `includes/AccountsRepository.php` (строки 1268–1269)
- **Проблема:** PHP-проверка `isset($existingLogins[$loginValue])` чувствительна к регистру, а MySQL collation — нет.
- **Фикс:** Нормализовать регистр (strtolower) перед проверкой.

## 3. MEDIUM — Средний приоритет {#3-medium}

### M-01. Рейс-кондишн в проверке timeout сессии
- **Файл:** `auth.php` (строки 443–475)
- **Описание:** Между проверкой и обновлением `last_activity` параллельные запросы могут «проскочить».

### M-02. Loose comparison `==` вместо `===`
- **Файл:** `includes/SessionManager.php` (строка 70)
- **Описание:** `$_SERVER['SERVER_PORT'] == 443` — нужно `=== '443'` (порт приходит как строка).

### M-03. IP-спуфинг в rate limiter
- **Файл:** `includes/RateLimitMiddleware.php` (строка 51)
- **Описание:** Используется только `REMOTE_ADDR`, без учёта X-Forwarded-For за прокси.

### M-04. CSRF-токен логируется частично
- **Файл:** `empty_trash.php` (строки 62–63)
- **Описание:** Первые 20 символов CSRF пишутся в лог. Не нужно логировать даже часть.

### M-05. json_encode() без безопасных флагов в inline-скриптах
- **Файл:** `templates/favorites.php` (строка 262) и другие
- **Описание:** Не все вызовы используют `JSON_HEX_TAG | JSON_HEX_AMP` — риск выхода из `<script>`.

### M-06. Open Redirect в DashboardController
- **Файл:** `includes/DashboardController.php` (строка 66)
- **Описание:** `header("Location: $redirectUrl")` — URL собирается из GET-параметров без валидации.

### M-07. IN-клаузы при bulk import могут превысить max_allowed_packet
- **Файл:** `includes/AccountsRepository.php` (строки 1491–1552)
- **Описание:** При 50 000 записей IN(?) содержит 50 000 значений.
- **Фикс:** Разбивать на чанки по 1000.

### M-08. Кеш в Database.php не потокобезопасен
- **Файл:** `includes/Database.php` (строка 217)
- **Описание:** `array_slice()` при каждом eviction — O(n), и не защищён от конкурентного доступа.

### M-09. StatisticsService грузит всё в память
- **Файл:** `includes/StatisticsService.php` (строка 301)
- **Описание:** UNION-запрос загружает все результаты сразу, без пагинации.

### M-10. Validator: смешение throw и return bool
- **Файл:** `includes/Validator.php` (строки 202–213)
- **Описание:** `validateCsrfToken()` бросает исключение при пустом токене, но возвращает bool в остальных случаях.

### M-11. Race Condition при регистрации статуса
- **Файл:** `api/index.php` (строки 410–425)
- **Описание:** Check-then-act без транзакции/лока.

### M-12. Unicode BOM не удаляется корректно в CsvParser
- **Файл:** `includes/CsvParser.php` (строка 131)
- **Описание:** `"\u{FEFF}"` — некорректный PHP-литерал. Нужно `"\xEF\xBB\xBF"`.

### M-13. Отсутствие disable кнопок при async-операциях
- **Файлы:** `assets/js/trash.js`, `view.php`
- **Описание:** Пользователь может кликнуть несколько раз, вызвав дублирование операций.

### M-14. Incomplete sensitive fields list в AuditLogger
- **Файл:** `includes/AuditLogger.php` (строки 27–35)
- **Описание:** `password`, `token`, `cookies` защищены, но могут быть другие (api_key, secret и т.д.).

### M-15. `intval()` на нечисловых строках возвращает 0, а не фильтрует
- **Файл:** `includes/FilterBuilder.php` (строка 204)
- **Описание:** `array_filter(array_map('intval', $idsArray))` — "abc" → 0 → отфильтруется, но "0abc" → 0 → тоже.

## 4. LOW — Низкий приоритет / Качество кода {#4-low}

### L-01. Dead Code: поддержка PHP < 7.3
- **Файл:** `includes/SessionManager.php` (строка 36)

### L-02. Magic numbers вместо констант
- **Файлы:** Многие (лимиты 1000, 10000, 50, 1MB, 20MB разбросаны по коду)

### L-03. `time()` для cache busting — новый запрос каждую секунду
- **Файл:** `templates/dashboard.php` (строки 15–21)
- **Фикс:** Использовать хэш файла или версию.

### L-04. Hardcoded API paths в JS
- **Файл:** `assets/js/favorites.js` (строка 40)
- **Описание:** Абсолютный путь `/api/favorites` не будет работать в подпапках.

### L-05. Memory leak: MutationObserver не очищается при unload
- **Файл:** `assets/js/favorites.js` (строки 11–12, 215–216)

### L-06. `alert()` вместо `showToast()` для ошибок
- **Файл:** `view.php` (строка 664)

### L-07. Unused variable `$placeholders`
- **Файл:** `duplicate.php` (строка 52)

### L-08. Некоторые SESSION-операции не используют strict comparison
- **Файл:** Несколько файлов используют `==` вместо `===` для проверки session-данных.

### L-09. console.error может раскрыть sensitive данные
- **Файл:** `assets/js/trash.js` (строки 193, 266)

### L-10. Непоследовательный нейминг параметров API
- **Описание:** `select_all` vs `selectAll` vs `select` — нет единого стиля.
