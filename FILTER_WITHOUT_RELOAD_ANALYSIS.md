# Анализ: обновление таблицы без полной перезагрузки страницы

## Текущее поведение

### Что уже работает без перезагрузки
- **refresh.php** — API возвращает JSON: `rows`, `totals`, `byStatus`, `filteredTotal`, `page`, `pages`, `columns`.
- **dashboard-refresh.js** — `refreshDashboardData()` делает `fetch('refresh.php?' + params)`, обновляет карточки статистики, вызывает `tableModule.updateRows(data)` и пагинацию.
- **table-module.js** — `updateRows(data)` перерисовывает только `tbody` по данным из JSON.
- Часть фильтров уже вызывает `debouncedRefreshDashboardData()` при изменении (поиск, статус marketplace, статус RK, слайдеры и т.д.) — таблица и статистика обновляются без перезагрузки.

### Где происходит полная перезагрузка

1. **Выбор статуса (чекбоксы в dropdown «Статус»)**  
   В **filters-modern.js** (стр. ~250–257) при `change` на `.status-checkbox` вызывается **`filtersForm.submit()`**.  
   При программном вызове `form.submit()` браузер **не генерирует событие `submit`**, поэтому `e.preventDefault()` в init-script не срабатывает. В результате выполняется обычная отправка формы (GET на текущую страницу) → **полная перезагрузка**.

2. **Удаление фильтра через chip (крестик)**  
   В **filters-modern.js** функции `removeFilterChip()` и `removeStatusChip()` делают **`window.location.href = url.toString()`** → полная перезагрузка.

3. **Кнопка «Обновить» (applyFiltersBtn)**  
   Кнопка имеет `type="submit"`. При клике пользователем срабатывает событие `submit` и `preventDefault()` в init-script, поэтому перезагрузки нет, но и явного применения фильтров через AJAX нет — кнопка по сути ничего не делает после отмены отправки.

---

## Техническая цель

Сделать так, чтобы при любом изменении фильтров (в т.ч. выбор/снятие статусов и удаление chip’ов) обновлялись только:
- URL (через `history.replaceState`),
- таблица (через `refreshDashboardData()`),
- карточки статистики и пагинация (уже внутри `refreshDashboardData()`),

без полной перезагрузки страницы.

---

## Логика и пользовательский сценарий

- Пользователь выбирает один или несколько статусов (или снимает выбор), удаляет фильтр по chip’у или меняет другие параметры.
- Ожидание: без мигания страницы обновляется таблица и счётчики, URL отражает текущие фильтры (можно шарить ссылку, обновлять страницу с теми же фильтрами).
- Текущее поведение: при выборе статуса и при удалении chip’а страница перезагружается целиком.

---

## Предлагаемые изменения

### 1. filters-modern.js — чекбоксы статусов

**Сейчас:** при `change` на `.status-checkbox` вызывается `filtersForm.submit()`.

**Нужно:** не вызывать `submit()`. Вместо этого:
- собрать выбранные статусы из чекбоксов (в т.ч. «Пустой статус»),
- построить URL с `status[]` и при необходимости `empty_status=1`,
- выполнить `history.replaceState(..., url)` и `refreshDashboardData()`.

Чтобы не дублировать логику, можно вызывать существующую функцию применения фильтра по статусам из init-script (например, `applyStatusFilter()`), если она доступна глобально, либо реализовать в filters-modern.js единую функцию «применить текущее состояние формы без перезагрузки» (построить query из формы → обновить URL → `refreshDashboardData()`).

### 2. filters-modern.js — removeFilterChip / removeStatusChip

**Сейчас:** после построения нового URL выполняется `window.location.href = url.toString()`.

**Нужно:** для того же URL выполнить `history.replaceState(null, '', url)` и `refreshDashboardData()`, без присвоения `location.href`.

### 3. Кнопка «Обновить» (applyFiltersBtn)

**Сейчас:** только `preventDefault()` на `submit`, явного применения фильтров нет.

**Нужно:** по клику (или по submit с preventDefault) собирать все параметры с формы (поиск, статусы, быстрые фильтры, диапазоны и т.д.), строить URL, делать `history.replaceState` и вызывать `refreshDashboardData()`. Тогда кнопка будет явно «применять» текущие значения полей без перезагрузки.

### 4. Синхронизация с init-script

- В init-script уже есть `applyStatusFilter()` и `debouncedRefreshDashboardData()`. Имеет смысл, чтобы применение по статусам (из filters-modern или из init-script) в итоге вызывало одну и ту же цепочку: обновление URL + `refreshDashboardData()`.
- Если «применить фильтры» из формы будет одна общая функция (построить query из формы → URL → replaceState → refreshDashboardData), то и чекбоксы статусов, и кнопка «Обновить», и (при желании) другие поля могут её использовать.

## Зависимости и риски

- **refresh.php** уже отдаёт данные по GET-параметрам; текущая разметка таблицы и `tableModule.updateRows()` с этим совместимы.
- **Пагинация** в init-script уже перехватывается и вызывает `refreshDashboardData()` без перезагрузки — менять не нужно.
- Нужно убедиться, что порядок скриптов (init-script, filters-modern, dashboard-refresh) не ломает наличие `refreshDashboardData` и опционально `applyStatusFilter` в момент вызова из filters-modern.js.
- После перехода на «только AJAX» для применения фильтров стоит проверить сценарии: только статусы, статусы + поиск, удаление chip’а, кнопка «Обновить».

---

## Итог

Чтобы при выборе статуса (и других фильтров) страница не перезагружалась, а обновлялась только таблица и статистика, достаточно:

1. Убрать `filtersForm.submit()` при изменении статусов и заменить его на обновление URL + `refreshDashboardData()`.
2. В `removeFilterChip` и `removeStatusChip` заменить `location.href` на `history.replaceState` + `refreshDashboardData()`.
3. Для кнопки «Обновить» реализовать применение текущих значений формы через ту же схему (URL + `refreshDashboardData()`).

Остальная инфраструктура (refresh.php, updateRows, пагинация) уже рассчитана на работу без полной перезагрузки страницы.
