# Отчёт: как ускорить проект и увеличить FPS ещё больше

После внедрения [FPS_MAXIMIZATION_PLAN.md](FPS_MAXIMIZATION_PLAN.md) ниже — дополнительные меры по ускорению и повышению плавности (FPS).

---

## 1. Что уже сделано (кратко)

- Sticky scrollbar только при видимости таблицы (IntersectionObserver).
- Один делегированный обработчик кликов.
- Content-visibility и containment для блока таблицы и `#tableWrap`.
- Порог виртуализации 45 строк, per_page > 50.
- Throttle ResizeObserver в sticky-scrollbar.
- Упрощённые transition и анимации в CSS.
- Шрифт Inter только 400 и 600.
- Режим `dashboard-perf-light` по `prefers-reduced-motion` и localStorage.
- Виртуализация «с первого кадра» при клиентском обновлении таблицы.

---

## 2. JavaScript: дальнейшие оптимизации

### 2.1 Один общий resize вместо двух

**Сейчас:** В `dashboard-init.js` два обработчика `resize`: `optimizedResizeHandler` и `optimizedResizeHandler2` (оба с RAF). Каждый при resize планирует свой кадр.

**Рекомендация:** Объединить в один обработчик, который за один RAF вызывает:
- `updateStickyScrollbar` (если нужно),
- `adjustTableDensity()`,
- `syncHeaderWidths()`.

**Эффект:** Меньше вызовов при resize, один проход по layout.  
**Файлы:** `assets/js/dashboard-init.js`.

---

### 2.2 Кнопка «Наверх»: не трогать layout в scroll

**Сейчас:** При скролле вызывается `toggleScrollToTop` с RAF и проверкой видимости кнопки (через класс show/hide). Убедиться, что внутри колбэка нет лишних чтений (getBoundingClientRect и т.п.) в горячем пути.

**Рекомендация:** Оставить только добавление/удаление класса по порогу `window.scrollY` (без чтения layout). Если уже так — зафиксировать в коде комментарием «без layout-чтений в scroll».

**Эффект:** Стабильный FPS при скролле страницы.

---

### 2.3 RowIdsCache при виртуализации «из данных»

**Сейчас:** В `dashboard-selection.js` `RowIdsCache.get()` при включённой виртуализации использует `tableVirtualization.allRows`. В режиме `enableFromData` массив `allRows` пуст, виртуализация хранит данные в `rowsData`.

**Рекомендация:** В `RowIdsCache.get()` при `tableVirtualization.enabled && tableVirtualization.rowsData.length > 0` строить список ID из `rowsData` (например, `row.id`), а не из `allRows` (DOM). Иначе «выделить все на странице» будет учитывать только видимые строки.

**Эффект:** Корректная работа выбора при виртуализации с первого кадра.  
**Файлы:** `assets/js/modules/dashboard-selection.js`.

---

### 2.4 Делегирование для чипов фильтров и dropdown

**Сейчас:** В `filters-modern.js` часть логики может вешаться на много элементов (чипы, кнопки удаления, пункты выпадающих списков).

**Рекомендация:** По возможности один `document` или контейнер-фильтров обработчик клика с `e.target.closest('.filter-chip-remove')`, `.filter-chip`, `.status-checkbox-item` и т.д., чтобы не плодить десятки listener’ов при большом числе фильтров.

**Эффект:** Меньше подписок, меньше работы при инициализации и при частых обновлениях DOM фильтров.

---

### 2.5 Рендер строк таблицы: шаблоны и DocumentFragment

**Сейчас:** В виртуализации «из данных» для каждого видимого индекса вызывается `renderRowFn(this.rowsData[i], this.columnKeys)`, результат вставляется через `innerHTML` во временный `<table><tbody>` и один `tr` переносится в fragment.

**Рекомендация (опционально):**
- Кешировать HTML для строк с одинаковой «структурой» (например, пустые ячейки), если такие строки повторяются.
- Для пачки строк за один раз использовать один `DocumentFragment` и один `insertBefore`, не трогать `innerHTML` контейнера по одной строке.

**Эффект:** Чуть меньше работы при скролле и смене видимого диапазона.  
**Файлы:** `assets/js/table-module.js`.

---

### 2.6 requestIdleCallback для некритичных задач

**Сейчас:** После refresh выполняются обновление счётчиков, избранного, колонок и т.д. Часть из них не блокирует отрисовку первого кадра.

**Рекомендация:** Обёртки в `requestIdleCallback` (с fallback на `setTimeout(..., 0)`) для:
- обновления подсказки виртуализации;
- сохранения выбранных ID в localStorage;
- любых «косметических» обновлений, не влияющих на то, что пользователь видит первым.

**Эффект:** Меньше конкуренции за главный поток в момент появления таблицы.  
**Файлы:** `assets/js/modules/dashboard-refresh.js`, `assets/js/modules/dashboard-selection.js`.

## 3. CSS: ещё меньше перерисовок

### 3.1 Точечное использование will-change

**Рекомендация:** Добавлять `will-change` только для элементов, которые реально анимируются в данный момент (например, открывающаяся модалка, появление toast), и снимать после анимации, чтобы не держать слои композитинга постоянно.

Пример:
```css
.toast.show { will-change: transform; }
.toast.hide { will-change: auto; }
```

**Не делать:** Глобальный `will-change` на скроллящемся контейнере или на всех карточках — это увеличивает потребление памяти и может снизить FPS.

**Файлы:** `assets/css/core-plugins.css`, точечно в компонентах с анимацией.

---

### 3.2 Упрощение тяжёлых селекторов

**Рекомендация:** В стилях избегать длинных цепочек и универсальных селекторов в горячих путях (таблица, карточки, скролл). Например, предпочитать `.ac-cell` вместо `table tbody tr td.ac-cell` там, где это не ломает специфичность.

**Эффект:** Быстрее пересчёт стилей при большом DOM.

---

### 3.3 Режим dashboard-perf-light: отключить сложные градиенты

**Рекомендация:** В `body.dashboard-perf-light` для карточек и заголовков заменять `linear-gradient` на сплошной цвет (например, `background: var(--bg-secondary)`), чтобы снизить стоимость отрисовки.

**Файлы:** `assets/css/core-base.css`.

---

## 4. Загрузка страницы и скрипты

### 4.1 Отложенная загрузка некритичных скриптов

**Сейчас:** Много скриптов подключается синхронно в конце `<body>` (Bootstrap, nouislider, все модули дашборда). Браузер парсит и выполняет их до отрисовки.

**Рекомендация:**
- Критичные для первого кадра: dom-cache, logger, dashboard-refresh, dashboard-selection, table-module, sticky-scrollbar, init-script (dashboard-init.js).
- С атрибутом `defer` или загрузкой после `DOMContentLoaded`: nouislider, dashboard-upload, saved-filters, части filters-modern (если не нужны до первого отображения таблицы).
- Один объединённый минифицированный бандл для продакшена вместо 15+ отдельных файлов уменьшит число запросов и парсинга.

**Эффект:** Быстрее First Contentful Paint и интерактивность.  
**Файлы:** `templates/dashboard.php`, при необходимости `build_assets.php`.

---

### 4.2 Preload для критичных ресурсов

**Рекомендация:** В `<head>` добавить:
```html
<link rel="preload" href="assets/js/dashboard-init.js" as="script">
<link rel="preload" href="assets/css/core-base.css" as="style">
```
чтобы браузер раньше начал загрузку блокирующих ресурсов.

**Файлы:** `templates/dashboard.php`.

---

### 4.3 Убрать или заменить ?v=<?= time() ?> в продакшене

**Сейчас:** Для скриптов и стилей используется `?v=<?= time() ?>`, что отключает кеш браузера при каждом заходе.

**Рекомендация:** В продакшене использовать версию по хешу файла или номеру релиза (например, `?v=<?= ASSETS_VERSION ?>`), чтобы кеш работал, а обновления подхватывались после деплоя.

**Эффект:** Меньше запросов и быстрее повторные загрузки.  
**Файлы:** `config.php`, шаблоны.

## 5. Сеть и API

### 5.1 Режим «light» refresh и сжатие

**Сейчас:** Уже есть опция `light` в `refreshDashboardData`, которая не показывает прелоадер и, возможно, уменьшает объём ответа.

**Рекомендация:**
- На бэкенде для `light` отдавать только необходимые поля (id, колонки для таблицы, счётчики), без лишних данных для карточек, если они не обновляются в этом запросе.
- Включить сжатие (gzip/brotli) для `refresh.php` и API.

**Эффект:** Меньше трафика и быстрее ответ после смены фильтров/пагинации.

---

### 5.2 Пагинация и кеш на клиенте (опционально)

**Рекомендация (сложность высокая):** Для недавно открытых страниц пагинации кешировать ответы (например, в памяти по ключу URL) и при быстром переключении «назад» показывать кеш, затем при необходимости дозапрашивать в фоне. Требует аккуратной инвалидации при смене фильтров.

**Эффект:** Мгновенное переключение между недавно открытыми страницами.

---

## 6. Измерения и мониторинг

### 6.1 Замер FPS и времени до интерактивности

**Рекомендация:** Включить в dev-режиме или по флагу простой замер:
- FPS при скролле (подсчёт кадров за 1–2 секунды через `requestAnimationFrame`).
- Время от начала загрузки до момента, когда таблица отрисована и виртуализация активна (по маркерам Performance API или `performance.now()`).

Так можно проверять эффект от каждой оптимизации и регрессии.

**Файлы:** `assets/js/core/performance.js` или отдельный маленький модуль.

---

### 6.2 Маркеры Performance API

**Рекомендация:** Обернуть ключевые этапы в `performance.mark()` / `performance.measure()`:
- начало и конец `refreshDashboardData`;
- начало и конец `updateRows` / первого виртуального рендера.

В DevTools → Performance будет проще находить узкие места.

## 8. Итог

Уже внедрённые меры сильно снижают нагрузку на рендер и главный поток. Дальше наибольший выигрыш дают:

- **Исправление RowIdsCache** при виртуализации из данных.
- **Один resize** и **requestIdleCallback** для некритичных обновлений.
- **Загрузка и кеш:** preload, defer части скриптов, версионирование ассетов.
- **CSS:** точечный will-change, упрощение в режиме perf-light.
- **Сеть:** облегчённый ответ light refresh и сжатие.

После каждого изменения полезно замерять FPS и время до интерактивности, чтобы подтверждать эффект и не допускать регрессий.
