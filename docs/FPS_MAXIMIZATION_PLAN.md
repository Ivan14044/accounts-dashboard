# План максимизации FPS панели

Цель: довести плавность интерфейса до максимума за счёт снижения нагрузки на рендер и скрипты.

---

## Приоритет 1 — Высокий эффект, разумные трудозатраты

### 1.1 Не вызывать sticky scrollbar при скролле выше таблицы

**Проблема:** При любом скролле страницы вызывается `optimizedUpdateStickyHScroll` → `window.updateStickyScrollbar()`, который читает `scrollWidth`, `clientWidth`, `scrollLeft` у таблицы. Когда пользователь скроллит карточки/фильтры, таблица ещё не в viewport, но layout читается.

**Решение:**
- Подписаться на блок таблицы через **IntersectionObserver** (например, `#accountsTableSection` или `#tableWrap`).
- Вызывать `updateStickyScrollbar()` только когда `entry.isIntersecting === true`.
- В `optimizedUpdateStickyHScroll` перед вызовом `updateStickyScrollbar()` проверять флаг «таблица в зоне видимости» и не вызывать обновление, если таблица не видна.

**Файлы:** `assets/js/dashboard-init.js` (место регистрации scroll), при необходимости `assets/js/sticky-scrollbar.js`.

---

### 1.2 Один делегированный обработчик кликов вместо многих

**Проблема:** В `dashboard-init.js` висит более 10 обработчиков `document.addEventListener('click', ...)`. Каждый клик вызывает все эти функции; в них — проверки по `e.target.closest(...)`. Лишние вызовы и проверки при каждом клике.

**Решение:**
- Ввести **один** глобальный обработчик `document.addEventListener('click', handleDocumentClick, { passive: false }` (passive: false только если где‑то нужен preventDefault).
- В `handleDocumentClick` по `e.target` или `e.target.closest(selector)` определять контекст (редактирование ячейки, копирование, чекбокс, кнопка «Наверх» и т.д.) и вызывать соответствующую логику (или делегировать в отдельные функции).
- Постепенно переносить логику из текущих обработчиков в эту единую точку и удалять старые подписки.

**Файлы:** `assets/js/dashboard-init.js`.

---

### 1.3 Content-visibility для блока таблицы

**Проблема:** Браузер перерисовывает и считает layout для всего дерева таблицы, даже когда блок далеко ниже viewport.

**Решение:**
- Для контейнера таблицы (например, `#accountsTableSection` или обёртки с `#tableWrap`) задать в CSS:
  ```css
  #accountsTableSection {
    content-visibility: auto;
    contain-intrinsic-size: auto 600px; /* примерная высота блока */
  }
  ```
- Когда блок уходит за пределы viewport, браузер может не рисовать его содержимое, что снижает нагрузку при скролле.

**Файлы:** `assets/css/core-tables.css` или отдельный блок в `templates/dashboard.php` (если стили страницы там).

**Осторожно:** Проверить, что пагинация и футер таблицы не «прыгают» при появлении блока (при необходимости скорректировать `contain-intrinsic-size` или обёртку).

---

### 1.4 Снизить порог включения виртуализации таблицы

**Проблема:** Виртуализация включается при `dataRows.length > threshold && per_page > 100`. При 50–80 строках на странице таблица рендерит все строки в DOM, при скролле внутри таблицы перерисовывается много ячеек.

**Решение:**
- Уменьшить `threshold` (сейчас, судя по коду, задаётся в options) до 40–50.
- Рассмотреть снижение порога по `per_page` (например, включать виртуализацию при `per_page > 50`), чтобы при 50 и 100 строках тоже использовалась виртуализация.

**Файлы:** `assets/js/table-module.js` (опции виртуализации и условие `shouldEnableVirtualization`).

## Приоритет 2 — Средний эффект

### 2.1 Throttle для ResizeObserver (sticky scrollbar)

**Проблема:** ResizeObserver для таблицы и `tableWrap` вызывает `update()` в каждом RAF при изменении размера. При ресайзе окна или изменении layout может быть серия быстрых вызовов.

**Решение:**
- В callback ResizeObserver не вызывать `update()` сразу, а ставить один отложенный вызов (например, через уже используемый `this.rafId` с отменой предыдущего) с небольшой задержкой (например, 50–80 ms) или только в следующем RAF, чтобы за один «тик» ресайза выполнялся один `update()`.

**Файлы:** `assets/js/sticky-scrollbar.js`.

---

### 2.2 Упростить MutationObserver для карточек и таблицы

**Проблема:** Несколько MutationObserver (карточки статистики, избранное, скрытие карточек) срабатывают при изменениях DOM. Если в колбэках тяжёлая логика или частые перерисовки, это может съедать FPS.

**Решение:**
- В колбэках MutationObserver только ставить флаг или пушить изменения в короткий список и обрабатывать их **одним** requestAnimationFrame или debounce 50–100 ms.
- Не делать в колбэке синхронные layout-чтения (offsetHeight, getBoundingClientRect и т.п.); переносить чтения в RAF или в отложенную обработку.

**Файлы:** `assets/js/modules/dashboard-stats.js`, `assets/js/modules/cards-hide-sync.js`, `assets/js/favorites.js`.

---

### 2.3 Ещё уменьшить тяжёлые тени и переходы в CSS

**Проблема:** В `core-plugins.css` и других файлах остаются `box-shadow`, `transition: all`, анимации (skeleton, pulse и т.д.), которые при большом количестве элементов дают лишнюю нагрузку.

**Решение:**
- Заменить оставшиеся `transition: all` на переходы только по конкретным свойствам (`background-color`, `opacity`, `border-color`).
- Заменить сложные или многослойные `box-shadow` на более простые (один слой, меньший blur).
- Для скелетонов и пульсирующих анимаций: сократить длительность или упростить до изменения только `opacity`.

**Файлы:** `assets/css/core-plugins.css`, при необходимости `assets/css/core-base.css`, `assets/css/core-theme.css`.

---

### 2.4 Шрифты: меньше начертаний

**Проблема:** Подключается Inter с весами 300;400;500;600;700;800 — много вариантов, дольше загрузка и разбор шрифтов.

**Решение:**
- Оставить только нужные начертания, например 400 и 600:  
  `family=Inter:wght@400;600&display=swap`
- Это ускорит первый рендер и немного снизит нагрузку при отрисовке текста.

**Файлы:** `templates/dashboard.php` (и другие шаблоны с тем же подключением Inter).

---

## Приоритет 3 — Дополнительно

### 3.1 Класс «лёгкий режим» по медиа или настройке

**Проблема:** На слабых устройствах или при желании пользователя можно ещё сильнее упростить интерфейс.

**Решение:**
- Ввести класс на `body`, например `dashboard-perf-light`, при установке которого (через настройку или `prefers-reduced-motion: reduce`):
  - отключить все анимации и свести transition к 0.01ms (как уже сделано для `prefers-reduced-motion`);
  - убрать оставшиеся box-shadow у карточек/таблицы или заменить на border;
  - при необходимости отключить sticky scrollbar или упростить его (только отображение, без ResizeObserver).
- Уже есть заготовка под `prefers-reduced-motion` в dashboard — расширить её и при желании привязать к явному переключателю «упрощённый режим».

**Файлы:** `assets/css/core-base.css`, при необходимости небольшой скрипт в `dashboard-init.js` или настройки в UI.

---

### 3.2 Виртуализация «с первого кадра» для больших списков

**Проблема:** Сейчас при большом `per_page` сначала рендерится полная таблица, затем включается виртуализация и DOM перестраивается. Возможен двойной тяжёлый рендер.

**Решение:**
- При загрузке данных, если `rows.length > threshold` и `per_page > N`, сразу рендерить только «виртуальный» слой (видимые строки + буфер), без полного innerHTML всех строк. Это потребует изменений в месте, где формируется HTML таблицы (например, в `dashboard-refresh.js` / table-module), чтобы при первом рендере передавать не полный список строк в DOM, а только диапазон для виртуализации.

**Файлы:** `assets/js/modules/dashboard-refresh.js`, `assets/js/table-module.js`, возможно бэкенд/шаблон, если разметка таблицы собирается на сервере.

---

### 3.3 Containment для контейнера скролла таблицы

**Проблема:** Область скролла таблицы может вызывать перерасчёт layout родительских блоков при прокрутке.

**Решение:**
- Добавить для `#tableWrap` (или контейнера с overflow: auto):
  ```css
  contain: strict;
  /* или */
  contain: layout paint;
  ```
- Это ограничит влияние layout/paint таблицы внутри контейнера. Проверить на вёрстке (особенно sticky-колонки и позиционирование).

**Файлы:** `assets/css/core-tables.css`.
